# Post-Pump Short Bot - 구현 명세 v1 (안정 운용 우선)

## 1) 목표
바이낸스 USDT 무기한 선물 전체 심볼(BTC, ETH 제외)에서,
7일 급등 후 급락 전환 구간을 숏으로 공략하는 자동매매 봇.

핵심 우선순위:
1. 멈추지 않고 안정적으로 동작
2. 포지션 노출 리스크 최소화
3. 장애 발생 시 자동 복구 및 즉시 알림

---

## 2) 실행 환경
- Python 3.11+
- 서버: Linux (Vultr Tokyo)
- 프로세스 관리: systemd (nohup 사용 금지)
- 단일 인스턴스 보장: `/tmp/pump_short_bot.lock`
- 시간 기준: UTC 고정

---

## 3) 거래/리스크 기본값
- 대상: 바이낸스 USDT 무기한 전체 심볼 (BTCUSDT, ETHUSDT 제외)
- 레버리지: 3x
- 마진 타입: ISOLATED
- 포지션 사이즈: 가용 잔고의 2% 명목가치 기준
- 최대 동시 포지션: 5개
- 동시 포지션 5개 도달 시 신규 신호: 즉시 스킵
- 동일 심볼 재진입: 포지션 청산 후 자동 재진입 금지 (다음 재스캔/조건 재충족 전까지)

---

## 4) 데이터 소스 및 아키텍처
안정성 우선 원칙:
- 실시간 감시: WebSocket 우선
  - mark price stream
  - kline stream (1m, 1h)
  - user data stream (주문/포지션 이벤트)
- REST는 초기 로드/복구/검증 용도로만 사용
- 심볼별 잦은 REST 폴링 금지 (Rate Limit 회피)

---

## 5) 전략 로직

### STEP 1. 펌프 스캔 (매 1시간)
- USDT 무기한 심볼 전체 조회
- BTC/ETH 제외
- 상장 7일 미만 심볼 제외
- UTC 일봉 기준 7일 상승률 계산
- 7일 상승률 >= 40% 심볼을 감시목록 등록/유지
- 스캔 완료 시 텔레그램 알림:
  - 감시목록 개수
  - 상승률 상위 5개

### STEP 2. 덤프 감지 (감시목록 대상 실시간)
조건은 순차적으로 적용:

1) 조건1 (필터)
- 최근 24개 1시간봉의 최고가(wick 기준) 계산
- 실시간 mark price 기준으로 비교
- `mark <= high_24h * 0.90` 이면 조건1 충족
- 충족 시점부터 60분 타이머 시작

2) 조건2 (트리거)
- 직전 마감 1분봉 기준
- `(open - close) / open >= 0.02` (음봉 바디 2% 이상)

3) 조건3 (트리거)
- 직전 마감 1분봉 거래량 >= 이전 20개 1분봉 평균의 4배

진입 조건:
- 조건1,2,3이 모두 충족되고
- 조건1 타임아웃(60분) 이전이며
- 동시 포지션 수 < 5

타임아웃:
- 조건1 충족 후 60분 내 조건2/3 미충족 시 해당 심볼 상태 리셋

### STEP 3. 진입
- 시장가 숏 주문
- 주문 체결 확인 후 즉시 트레일링 스탑 주문:
  - 주문 타입: `TRAILING_STOP_MARKET`
  - 방향: BUY (숏 포지션 청산)
  - `callbackRate = 10`
  - `activationPrice` 미지정 (즉시 활성화)
  - `reduceOnly = true`
- 진입 알림 전송:
  - 심볼, 체결가, 7일 상승률, 투입 명목가치

---

## 6) 트레일링/포지션 안정성 정책 (핵심)

### 6-1. 진입 직후 트레일링 생성 실패
- 1차 실패 시 재시도 (지수 백오프 1s, 2s, 4s)
- 총 3회 실패 시:
  - 즉시 시장가 청산
  - 긴급 텔레그램 알림 전송

### 6-2. 재시작 후 기존 포지션 복구
- 기존 오픈 숏 포지션 조회
- 각 포지션별 트레일링 존재 여부 확인
- 트레일링 미존재 시 재설정 시도 (1s, 2s, 4s)
- 재설정 최종 실패 시:
  - 즉시 시장가 청산(B 정책)
  - 긴급 텔레그램 알림

### 6-3. 포지션 종료 사유 분류
종료 감지 시 가능한 범주:
- `trail_hit`
- `manual_close`
- `forced_liquidation`
- `unknown`

`unknown`이면:
- 즉시 경고 알림
- 최근 주문/체결 raw 요약 포함

---

## 7) 감시목록 제거 조건
- 최근 6개 1시간봉 동안 신저가 갱신 없음 -> 감시목록에서 제거
- 제거 알림 전송
- 단, 이후 재스캔에서 7일 상승률 >= 40%면 재등록 가능

---

## 8) 주문 수량 계산 규칙
- 기준: 가용 잔고 * 2% (명목가치)
- 심볼별 exchange info 반영 필수:
  - `minQty`
  - `stepSize`
  - `minNotional`
- 반올림/내림 후 유효성 재검증
- 유효 최소수량 미달 시 해당 신호 스킵 + 사유 로그

---

## 9) 텔레그램

### 9-1. 보안
- 명령어 처리 시 `chat_id` 화이트리스트 검증
- 비허용 chat_id는 무시

### 9-2. 자동 알림
| 상황 | 내용 |
|---|---|
| 스캔 완료 | 감시목록 수, 상위 5개 심볼 |
| 진입 | 심볼, 진입가, 7일 상승률, 명목가치 |
| 종료 | 심볼, 종료 사유, 대략 PnL |
| 감시목록 제거 | 심볼, 제거 사유 |
| 장애/복구 | API 실패 누적, WS 재연결, 복구 실패, 강제청산 수행 |

### 9-3. 원격 명령어
| 명령어 | 기능 |
|---|---|
| /status | 잔고, 오픈 포지션 수, 감시 심볼 수, 헬스 상태 |
| /list | 감시목록 (상승률 순) |
| /positions | 오픈 포지션 목록 |
| /closeall | 전체 포지션 시장가 청산 |
| /close SYMBOL | 특정 심볼 시장가 청산 |
| /exclude SYMBOL | 영구 제외 등록 |
| /watchlist | 영구 제외 목록 조회 |
| /stop | 봇 정지 |
| /restart | 봇 재시작 |

## 10) 장애 대응 표준
- API 호출 실패: 지수 백오프 `1,2,4,8,16초`, 최대 5회
- 연속 실패 임계치(기본 10회) 초과 시:
  - 신규 진입 일시 중단
  - 텔레그램 경고
- 복구 확인 후 신규 진입 자동 재개
- WebSocket 끊김 시 자동 재연결 루프 및 재구독

## 14) 구현 노트
- 트레일링 10%는 진입가와 1:1 고정 손절이 아님
- 목표는 "정확히 10% 가격 일치"보다 "보호 주문 누락 방지"
- 안정성 최우선: 보호주문 실패 시 즉시 노출 포지션 제거
